## Durable repo and host rules

- `cli.main()` must resolve the project `repo_root` from the invoked entrypoint path or current workspace markers, not from `__file__`, because the packaged console script may run from `.venv/bin` while the module lives under `site-packages`.
- `sync_clients()` should preflight target config rendering before writing any managed target files, and target-removal cleanup depends on persisting target `config_path` / `skills_dir` in state so removed targets can have stale managed output cleaned on the next sync.
- Invalid OpenCode JSONC should surface as `SyncError`, and leading comments containing braces are valid input that must not break top-level key updates.
- `install_service()` must leave `WorkingDirectory=` unquoted in the generated systemd unit; quote `ExecStart` arguments as needed, and validate space-containing paths with `systemd-analyze verify`.
- CLI-side MCP add/remove rollback must restore both `shared-ai-config.json` and any already-written managed target files or state snapshots when sync fails after target writes begin.
- When reading pre-upgrade state that lacks `skills_dir`, skill cleanup for removed Codex/Claude targets should fall back to inferring the legacy skills directory from the target config location and the managed link names.
- If a managed prompt target is a symlink, cleanup for disabled or moved prompt outputs must remove both the symlink path and the resolved backing file unless the new target resolves to the same file.
- Legacy target cleanup must also handle pre-upgrade state that lacks `config_path`; infer conventional client config locations for Codex, Claude, and OpenCode before giving up on removing stale managed MCP entries.
- `shared-ai-config.json` now supports `${REPO_ROOT}` and `${HOME}` placeholders across path-like target fields and stdio MCP command/cwd/env values, so the shared config should stay checkout-relative and login-user-relative instead of hard-coding `/home/<user>` paths.
- Shared skill sync now mirrors only repo-local `skills/shared/`; client-specific repo skills live under `skills/<client>/`, and client-native system skills stay owned by each CLI instead of being cross-synced through this repository.
- Repo-local MCP wrappers under `tools/mcp/shared/` and `tools/mcp/<client>/` are the stable entrypoints for synced MCP commands and must bootstrap only vendored runtimes under `vendor/mcp/`; missing repo-local runtime state is a fail-fast error, not a reason to fall back to `${HOME}` installs.
- Shared-vs-specialized sync ownership is folder-based, not filter-based: shared repo skills stay as direct children under `skills/shared/`, client-specific repo skills live under `skills/<client>/`, shared MCP wrappers stay under `tools/mcp/shared/`, future client-specific wrappers should live under `tools/mcp/<client>/`, and future client-specific docs should live under `docs/<client>/`.
- The Serena chain is now repo-local end-to-end: `tools/mcp/shared/serena-manager.sh` installs vendored `vendor/mcp/serena-manager`, and its default `ManagerConfig` must invoke `tools/mcp/shared/serena-agent.sh`, which runs the vendored `vendor/mcp/serena-agent/pylib/serena` package instead of `/home/<user>/.local/bin/uvx`.
- Vendored MCP updates are now owned by `src/ai_config_sync/mcp_updates.py`: `serena-manager` is maintained directly in `vendor/mcp/serena-manager`, while `mcp-update-serena-agent`, `mcp-update-codegraph`, `mcp-update-node-repl-linux`, and `mcp-update-all` are the supported refresh entrypoints for the externally versioned vendored MCP runtimes.
- `mcp-update-all` must be transactional across all three externally versioned MCPs; if a later component update fails, earlier `serena-agent` / `codegraph` / `node-repl-linux` file changes must roll back to the pre-run state instead of leaving a partially updated vendor tree.
- `vendor/mcp/codegraph/package.json` and `vendor/mcp/codegraph/package-lock.json` should stay exact-version aligned; do not leave the manifest pinned while the lockfile still references a caret range.
- Repo-local MCP startup now has a separate preparation phase: `mcp-preflight` owns downloading the pinned toolchain from `toolchain.lock.json`, generating `vendor/toolchain/runtime-env.sh`, and rebuilding stale MCP runtimes. Runtime wrappers under `tools/mcp/shared/` and `tools/mcp/<client>/` must only read those prepared paths and fail fast instead of running `uv` / `npm` / `node` installs during startup.
- Vendored MCP update commands in `src/ai_config_sync/mcp_updates.py` must also prepare and consume the repo-local toolchain first; do not call host PATH `uv` / `npm` directly from update flows, because that breaks fresh or minimal hosts even when `mcp-preflight` already prepared the pinned toolchain.
- For the current Linux/x86_64 host, prefer `toolchain.lock.json` plus managed `vendor/toolchain` binaries over PATH lookups. Wrapper scripts should avoid `#!/usr/bin/env bash`, `dirname`, npm thin launchers, and other extra PATH-sensitive layers when an absolute repo-local path or embedded binary exists.
- OpenCode should no longer depend on `/usr/local/bin/opencode` or `/usr/local/lib/node_modules/opencode-ai` for daily use. Use `ai-config-sync` commands to install managed releases under `~/.local/share/ai-config-sync/opencode/releases/<version>`, keep `/home/admin101/.local/bin/opencode` as the stable wrapper, and run the system service `opencode-web.service` as the canonical boot path; the old user unit `opencode-web-managed.service` should stay disabled to avoid port conflicts.
- Claude Code should use the native user-level install under `~/.local/bin/claude` and its built-in `claude update` / `claude install <version> --force` flow. Do not keep a parallel global npm `@anthropic-ai/claude-code` install on this host, because the native updater reports multi-install drift and the effective runtime should stay under `~/.local/share/claude/versions/`.
- `pi-web` upstream resolves the `@earendil-works/pi-ai` OAuth helper through a brittle module-root probe. On this Linux host the user-level `pi` install lives under `~/.local/lib/node_modules/@earendil-works/pi-coding-agent`, so managed `pi-web` installs must provide a compatibility path under `~/.npm-global/lib/node_modules/@earendil-works/pi-coding-agent` and should set `PI_AI_OAUTH_INDEX` in the systemd unit to the real `node_modules/@earendil-works/pi-ai/dist/utils/oauth/index.js` path.
- The web dashboard should treat `claude` and `opencode` card activity as runtime-process detection first, not only `systemctl`, because this host commonly runs live CLI sessions without the matching managed service being active. The dashboard also needs the real Claude updater path `scripts/claude/update-claude.sh`, and `pi --version` may emit a bare semver string such as `0.80.2`, which the status parser must accept.

- Added repo-managed shared skills under `skills/shared/`: `paseo`, `paseo-handoff`, `paseo-advisor`, `paseo-loop`, `paseo-committee`.
