- Dashboard tool runtime status in `src/ai_config_sync/web_server.py` should come from live process detection, not only systemd service state. `claude`, `codex`, `opencode`, and `pi` can all report `status=active` from runtime processes even when they are not systemd-managed services.
- Only `opencode` is a managed service in the current dashboard model. Tool-card stop/start actions should therefore be service-level actions for `opencode-web.service`, while other process termination remains in the process manager.
- OpenCode skill counts do not come from a skills directory. They should be read from `~/.config/opencode/opencode.jsonc` `agent` keys using the `skill-` prefix. Codex/Claude/Pi skill counts come from skill directories that contain `SKILL.md`.
- MCP inventory/details should be built from resolved sync config via `load_sync_config(...)`, not from raw `shared-ai-config.json`, so wrapper paths/commands reflect concrete runtime values.
- OpenCode latest-version lookup can hit GitHub API 403 on this host. Use the API first, then fall back to parsing the redirect target from `https://github.com/anomalyco/opencode/releases/latest`.
- Serena project grouping in the dashboard should be sourced from `vendor/mcp/serena-manager/state/*/meta.json`, because those `meta.json` files are the reliable place that records `project_root`, endpoint, and active agent PID. Live launcher processes alone do not carry enough project metadata.
- Pi managed component inventory comes from `inspect_pi_packages(...)` over `~/.pi/agent/npm`. Per-component latest versions should be resolved via npm registry lookups and shown beside installed versions so single-package upgrades can be targeted from the UI.
- Generic MCP project grouping should use best-effort project hints from process command flags such as `--project` / `--path`, then fall back to parent process `cwd`. When the parent cwd is only the user home and no explicit project hint exists, group that session as an unmapped owner session instead of fabricating a project path.
- Paseo version reporting should distinguish at least three sources: CLI version (`cliVersion`), daemon/server package version (read from installed `@getpaseo/server` package when `daemonVersion` is missing), and the latest observed Desktop client version parsed from recent daemon log user-agent entries. These are different components and can legitimately differ.
