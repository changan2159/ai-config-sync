# Codex-Specific Additions

- When referencing local files in responses, prefer Markdown links so the user can click them in the Codex chat UI.
- When a client-specific line number format matters, prefer standard file references such as `[file.cs](F:/ProjectFile/repo/src/file.cs):120` or `[file.cs](F:/ProjectFile/repo/src/file.cs#L120)`.
- In Python, pass `encoding="utf-8"` when reading or writing text files that Codex creates or modifies.
- When Serena is available, prefer it for brownfield codebase navigation in symbol-friendly languages and repos where symbol-aware structure is reliable: cross-file tracing, declaration and reference lookup, ownership discovery, call-chain inspection, and other cases where text search alone is likely to miss the real seam. This commonly applies to C#, TypeScript, Java, Python, Go, and similar LSP-friendly languages. Do not force Serena for weak-symbol or poorly indexed codebases, simple single-file edits, straightforward `rg` text search, build or test execution, or small frontend styling changes where symbolic navigation adds little value.
- When CodeGraph is available, use it as a fast candidate-discovery layer for indexed source code: project structure, related symbols, likely owner files, callers/callees, and impact sets. Treat CodeGraph as a discovery index, not the final source of truth; confirm edit targets and ambiguous flows with Serena or direct file reads before changing code. Prefer explicit symbol/file/route/DTO anchors over broad natural-language queries, and expect manual `codegraph sync` in sessions where watch is intentionally disabled.
- Default retrieval sequence when both Serena and CodeGraph are available: start with `CodeGraph` for anchored candidate discovery, switch to `Serena` to confirm owners/symbols/call chains, and use `rg` for strings, config keys, routes, JSON fields, SQL, logs, and non-indexed assets. If the first `CodeGraph` query is broad or noisy, immediately tighten it with explicit anchors such as symbol names, endpoint paths, DTO names, file stems, or modality/protocol terms instead of asking another broad natural-language question. For implementation work, do not edit from CodeGraph output alone; verify the final target with Serena or direct reads first. When the user is asking about the current repository and the core job is retrieval or structure understanding, prefer `fast-codebase-retrieval` as the default skill path.
- For Linux-hosted Codex sessions on this machine, follow the approved Serena and `.NET` environment workflow in `/home/admin101/.codex/rules/linux-serena-dotnet-host-notes.md`. These host notes are validated on the current Ubuntu machine and should be re-checked before copying distro-specific paths to another Linux host. If Serena disappears or `.NET` tooling drifts, repair that environment instead of inventing alternate paths.
- If a repository uses `.NET` or EF Core, follow `/home/admin101/.codex/rules/dotnet-workflow-defaults.md` for CLI tool, restore, and migration defaults instead of improvising per-project behavior.
- For persistent Codex work, prefer repository-local `AGENTS.md`, project docs, and Serena memories for durable context according to their scope.

# Codex Skill Routing Defaults

- Treat these as default routing hints; repository-local `AGENTS.md` or explicit user instructions can narrow, override, or forbid specific skills.
- For bug reports, failing tests, build breaks, runtime exceptions, or behavior that is reproduced but not yet explained, prefer `systematic-debugging`.
- For production behavior changes where tests are practical, prefer `test-driven-development`.
- For non-trivial brownfield work in symbol-friendly languages, prefer Serena-driven routing such as `serena-workflow` or a language-specific symbolic workflow before broad text-only exploration.
- For requests centered on the current repository where the main ask is to find implementation locations, related files, owners, call chains, frontend/backend linkage, project structure, or the smallest complete cross-file context, prefer `fast-codebase-retrieval` before broader implementation or refactor skills. Combine it with `serena-workflow` when symbol confirmation is important.
- For rule, docs, memory, glossary, or durable-context updates, prefer `agents-self-evolution`.
- For general code review, regression review, or second-pass verification, prefer `code-review`; use `security-review` instead when security is the primary concern.
- For product-style frontend implementation, redesign, polish, responsive fixes, or in-repo UI improvement work, prefer `frontend-design`. Pair it with `frontend-ui-engineering` when component logic, state flow, or shared UI architecture is a substantial part of the task.
- Prefer the `Product Design` plugin flow only when the user explicitly invokes that plugin, or when the task is design-first rather than code-first: visual ideation, prototype exploration, URL cloning, screenshot or Figma matching, image-to-code work, flow audit, or brief-led concept exploration. In those cases, route through the plugin's context gate first instead of jumping straight into code edits.

# Codex Verification Additions

- For moderately complex or higher changes, default to a second-pass review. Prefer a child verifier agent first when available, then the `code-review` skill or equivalent read-only reviewer, and fall back to a stricter manual `git diff` review only when no independent verifier is available. Address review findings before final delivery.
- When using Claude Code CLI or another external reviewer for cross-review, feed a fixed commit or exported patch plus the smallest relevant surrounding context (callers, callees, tests, and config). Do not rely on raw diff alone for cross-module or behavioral changes, and keep the reviewer read-only.
