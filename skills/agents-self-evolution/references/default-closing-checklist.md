# 默认收尾沉淀清单

当任务属于非一次性的编码、调试、评审、重构、排障、架构理解或文档补全时，结束前按此清单执行。

## 1. 先判断是否值得沉淀

只要满足任一项，就进入沉淀流程：

- 发现了稳定的项目结构、模块边界、注册点、入口点、测试路径
- 发现了已验证业务规则、领域术语、工作流
- 验证了命令、构建约束、测试约束、环境要求
- 用户重复纠正了同一类行为、表述或偏好
- 若不记录，后续代理大概率会再次犯同样错误

若以上都不满足，不需要为了“有记录”而强行写文档。

## 2. 给知识分类

从以下类别中选最贴切的一类或两类：

- 项目结构 / 模块边界 / 代码归属
- 业务规则 / 领域理解
- 术语 / 别名 / 易混淆概念
- 业务或技术工作流
- 已验证命令 / 构建测试约束
- 用户偏好 / 重复纠错
- 候选规则 / 未确认问题

## 3. 选择最小正确落点

优先选择最窄、最合适的存储位置：

- `AGENTS.md`
  - 适合：简短规则、默认行为、索引入口、禁止事项
  - 不适合：长篇业务逻辑或大量细节
- `docs/ai-context/project-structure.md`
  - 适合：入口点、模块边界、依赖方向、注册点、测试布局
- `docs/ai-context/business-domain.md`
  - 适合：已验证业务行为、领域规则、不变量、禁止假设
- `docs/ai-context/glossary.md`
  - 适合：术语、别名、易混淆概念
- `docs/ai-context/workflows.md`
  - 适合：业务流程、技术流程、状态转换、参与者
- `docs/ai-context/open-questions.md`
  - 适合：尚未确认的问题、低置信度推断
- Serena memories
  - 适合：动态纠错、观察、候选规则、演进日志、临时上下文

## 4. 证据分级

- `high`：用户明确说明，或代码 / 测试 / schema / 文档直接支持
- `medium`：多处迹象一致，但仍有少量未验证部分
- `low`：主要依赖推断，尚未确认

规则：

- `high` / `medium` 才能进入稳定文档
- `low` 优先写入 `open-questions.md` 或 Serena memories
- 不要把推断直接升级成权威规则

## 5. 控制噪音

满足以下任一情况，优先不写或只写 Serena：

- 只是一次性实现细节，对后续任务几乎没有复用价值
- 只是局部变量级、单函数级细节，不影响项目理解
- 与已有文档重复，只是换一种说法
- 证据太弱，写入后反而会误导

## 6. 最小变更原则

- 优先补充现有文档，不轻易新建平行文件
- 一次只写最小必要事实
- 若已有冲突，不静默覆盖；改写为候选说明或开放问题
- 中文团队优先中文说明，但保留路径、类名、方法名、配置键、命令等英文锚点

## 7. 收尾输出

最终回复只报告真正有意义的沉淀结果，例如：

- 更新了哪些规则 / 文档 / memory
- 为什么这次值得记录
- 若仍有不确定项，明确写成开放问题

不要把每个细枝末节都当成“沉淀成果”。
